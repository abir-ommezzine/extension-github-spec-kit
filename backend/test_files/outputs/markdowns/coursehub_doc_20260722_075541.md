# CourseHub - Technical Specification & Architecture Document

## 1. Executive Summary & Architecture Overview

### 1.1 Executive Brief
CourseHub is a course management platform providing a RESTful API (v1) for instructors and students. The system implements a role-based access model focused on course lifecycle management and student enrollment tracking, utilizing a JWT-based authentication pattern and a standardized response envelope for all API interactions.

### 1.2 Maturity Assessment
The specification provides a high level of technical detail regarding API contracts and business logic, yet it is categorized as REFINEMENT. While the implementation details are granular, the project lacks a strategic foundation, specifically missing high-level business goals, scope boundaries, and a defined list of out-of-scope features, which may lead to scope creep during execution.

### 1.3 Technical Stack
* **Authentication**: JWT
* **Email Service**: Resend
* **Data Format**: JSON
* **Password Hashing**: bcrypt
* **Identifiers**: UUID

### 1.4 Architectural Constraints
* **Base URL**: `http://localhost:8000/api/v1`
* **Content-Type**: `application/json`
* **Token Expiry**: Access token: 15 minutes / Refresh token: 7 days
* **Course Deletion**: Forbidden if active enrollments exist (`ActiveEnrollmentConstraint`)
* **Enrollment Logic**: Requires published status of the target course; duplicate enrollment for the same student is forbidden.
* **Auto-completion**: Triggered when `progress == 100` and `completed_at` is null.
* **Access Control**: Instructors can only manage/delete courses they own; Students can only access/update their own enrollments.

### 1.5 Critical Dependencies
* **Resend**: Async welcome email service.
* **JWT Bearer token**: Core authentication flow.
* **Referential Integrity**: Strict foreign key dependence of Enrollment on User and Course entities.
* **Cascade Dependency**: Modules are strictly dependent on Course entities.
* **Integrity Constraint**: Hard link between Course deletion and Enrollment records.

## 3. Detailed Technical Specifications & Business Rules

### 3.1 Requirements Traceability
| ID | Requirement Type | Description | Source Section |
| :--- | :--- | :--- | :--- |
| **API-AUTH-JWT** | Non-Functional | Authentication via JWT Bearer tokens in Authorization header. | API Overview |
| **API-ENVELOPE** | Non-Functional | All responses must follow a consistent envelope containing data, meta, and errors. | Response Envelope |
| **AUTH-REG** | Functional | Students can self-register and will receive an asynchronous welcome email via Resend. | POST `/api/v1/auth/register` |
| **AUTH-REFRESH** | Functional | Access tokens expire in 15 minutes; refresh tokens expire in 7 days. | POST `/api/v1/auth/refresh` |
| **CRS-CREATE** | Functional | Instructors can create courses with associated modules. | POST `/api/v1/courses` |
| **CRS-DEL-CONSTRAINT** | Constraint | A course cannot be deleted if it has active enrollments (ActiveEnrollmentConstraint). | DELETE `/api/v1/courses/{course_id}` |
| **ENR-VALIDATION** | Functional | Enrollment requires the course to be published and forbids duplicate enrollment for the same student. | POST `/api/v1/enrollments` |
| **ENR-AUTOCOMP** | Functional | If progress is set to 100 and completed_at is null, completed_at is automatically set to current timestamp. | PUT `/api/v1/enrollments/{enrollment_id}` |

### 3.2 Security Rules
* **Authentication**: All protected endpoints require a `Authorization: Bearer <token>` header.
* **Authorization**: Role-based access control (RBAC) is enforced:
    * `Student`: Self-registration, enrollment management.
    * `Instructor`: Course creation, module management, ownership-based deletion.
* **Data Protection**: Passwords must be stored as bcrypt hashes and never returned in API responses.
